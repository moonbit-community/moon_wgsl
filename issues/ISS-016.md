## Description
Imported module declaration retention must follow the pinned `naga_oil` module construction model. The legacy tracker described cases where upstream emits declarations that are not directly reachable from the root entry point while moon_wgsl pruned them, which means compose was still optimizing rather than cloning upstream construction.

This issue is migrated from the legacy tracker row `WGSL-282`.

## Design
Use structured graph and IR data only. Retention must be driven by import request planning, generated-import provenance, module construction events, and root/source slice identity. Do not reintroduce name-based unqualified binding fallbacks or source-text reachability shortcuts.

The correct end state is not "keep more declarations"; it is "construct the same module graph as naga-oil, then emit from that graph."

## Acceptance Criteria
- [x] Imported declaration retention is owned by structured module/import construction data, not final generated names or source-text scans.
- [x] Selected-item slices retain naga-oil-required type, alias, const, override, function, and resource side effects without leaking unrelated helper-module resources.
- [x] `bash tools/check_external_naga_oil_compose_parity.sh` passes in strict writer/order/name mode with no writer drift.
- [x] Pipeline-interface subset parity remains valid for the full external Bevy compose corpus.
