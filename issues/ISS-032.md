## Description
`tools/check_architecture_guardrails.sh` is useful, but much of it is grep-based. It catches forbidden strings and known regressions, yet it is not a semantic proof of module ownership, public API boundaries, or issue/parity inventory freshness.

This makes the guardrail hard to maintain as the workspace grows.

## Design
Keep the shell script as the top-level aggregator, but move high-risk checks into structured tools or manifests. Prefer machine-readable inventories for parity cases, module public APIs, issue tracker state, and forbidden dependency edges.

## Acceptance Criteria
- [x] High-risk guardrails for module dependency edges and public API leakage use structured metadata instead of raw grep.
- [x] Parity inventory gates report executed case lists and compare them with manifests.
- [x] Issue tracker consistency is checked by the derived tracker tool or an explicit validation wrapper.
- [x] The shell guardrail remains as an orchestrator and documents which checks are semantic versus textual.
- [x] Existing forbidden-pattern regressions remain covered by tests or structured checks.
