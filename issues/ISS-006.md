## Description
The stronger representative trace added in ISS-003 shows that Moon function bodies are still not structurally equivalent to upstream Naga bodies. Moon lowers many local declarations and generated helper locals as `Statement::Declare(...)` nodes inside the body, while upstream Naga stores locals in the function local arena and uses body statements for expression `Emit` ranges, side effects, calls, stores, and control flow.

This mismatch explains why selected expression arena parity can pass while statement execution traces still drift. It also makes byte writer parity fragile because writer policy has to compensate for a body model that is not upstream-shaped.

## Design
Refactor lowering and writer body planning so the Naga-compatible path has a single Naga-style function body model:

- locals live in `Function.local_variables`, not as semantic body declaration statements;
- initialization dependencies are represented by upstream-style `Emit` ranges and `Store` statements where Naga would use them;
- control-flow conditions, call arguments, and store operands get the same pre-statement `Emit` range boundaries as upstream;
- writer-local declarations and helper temp names are derived from the Naga-style body arena, not from source-level declaration statements.

This should be a structural lowering/body-plan change, not a trace-script normalization or output-text patch.
