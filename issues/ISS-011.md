## Description
Short-circuit expressions currently have multiple paths: control-flow condition lowering, return expression materialization, expression-node materialization, and handle-based materialization. The `prepare_world_normal` drift shows that conversions such as `f32(!double_sided || is_front)` still expose ordering and local materialization differences.

## Design
Create `ShortCircuitPlanner` as the only module that decides short-circuit local allocation, condition inversion, branch value stores, result loads, and conversion-input behavior. Node-based and handle-based short-circuit paths should converge on the same plan.

## Acceptance Criteria
- [x] `LogicalAnd` and `LogicalOr` lower through one planner for condition, value, return, assignment RHS, and conversion input contexts.
- [x] `prepare_world_normal` focused trace passes without adding a local rule patch.
- [x] Existing `world_to_directional_light_local` short-circuit trace remains passing.
- [x] Redundant node-based or handle-based short-circuit helper branches are deleted.
