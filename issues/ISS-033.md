## Description
Issue tracker and parity documentation can drift from the actual repository state. Some closed workspace-split issues had unchecked acceptance criteria, and `docs/naga_oil-parity.md` still referenced stale release and parity status.

Manual cleanup is not enough; the repository needs a small freshness gate.

## Design
Use the markdown issue tracker derivation tool as the source for issue index state, and add lightweight checks that catch stale release references, stale parity counts, and closed issues with unchecked acceptance criteria unless explicitly annotated.

## Acceptance Criteria
- [x] Closed issues cannot retain unchecked acceptance criteria unless an intentional exception is recorded.
- [x] `issues/README.md` is always regenerated from `derive-tracker.wasm`.
- [x] `docs/naga_oil-parity.md` records the current release, compose parity counts, and Naga writer trace counts.
- [x] A CI or local guard detects stale parity counts or stale release references after publish/parity runs.
- [x] Documentation freshness checks run as part of the architecture/tooling gate.
