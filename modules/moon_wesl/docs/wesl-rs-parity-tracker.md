# moon_wesl vs wesl-rs Parity Tracker

Last updated: 2026-07-20

Baseline target for parity planning: the `wesl` 0.3.2 library behavior. The
standalone `wesl-cli` crate is outside this workspace's product boundary. This
tracker focuses on closing the largest library and architecture gaps in the
current `moon_wesl` implementation.

## Status Legend

- `TODO`: not started
- `IN_PROGRESS`: currently being fixed
- `BLOCKED`: blocked by platform/public API limitations or external dependency constraints
- `DONE`: fixed and verified locally

## Issues

| ID | Source | Problem | Status | Notes |
| --- | --- | --- | --- | --- |
| PARITY-001 | `wesl::Wesl`, `wgsl-parse`, local `compile.mbt` | Replace the current text-oriented compiler core with an AST-backed pipeline. | `IN_PROGRESS` | Phase 1. The work was reset on 2026-04-15 after concluding that incremental keyword-matching cleanups were not a viable long-term compiler architecture. Current slices landed: dedicated `error.mbt`, `syntax.mbt`, and `parser.mbt` files now define a compiler-facing `TranslationUnit` / `ImportStatement` / `GlobalDeclaration` layer; `compile.mbt` now consumes `parse_translation_unit(...)` output to build module indexes and import graphs, and the old inline top-level parsing entrypoints have been removed from `compile.mbt`; lowering now consumes parsed global declarations via `parse_global_declarations(...)` rather than reparsing emitted output ad hoc. Since the latest rewrite, `parser.mbt` is no longer a character-index parser: it tokenizes source first and drives import/declaration parsing from `ArrayView[ParserToken]` pattern matching. The newest slices also promote `syntax.mbt` from a lightweight record bag into a more structured syntax layer and start pulling core compiler behavior away from blind text substitution: attributes now keep names, argument text, and spans; imports are stored as recursive `ImportNode` trees and flattened later by syntax helpers; global declarations now carry typed headers such as `FunctionDeclaration`, `AliasDeclaration`, `ConstDeclaration`, and `VarDeclaration`; conditional translation can target non-block regions such as declarations, statements, parameters, and members; and identifier analysis/rewrite now tracks minimal lexical scope so local shadowing does not get rewritten as a global reference. Remaining gap in this issue: the parser is token-based and the syntax tree is materially richer, but it is still a lightweight hand-written grammar rather than a complete WGSL/WESL syntax tree on the scale of `wgsl-parse`, and later stages such as name resolution, emit, and lowering are still much closer to a textual compiler than to the full `wesl-rs` semantic pipeline. Validation: `moon check --target all` and `moon test -v` both pass locally on 2026-04-15; the expanded upstream suite is now 25/25 green with active import and conditional-translation parity coverage. Exit criteria: imports, re-exports, condcomp, strip, and emit operate on parsed nodes throughout, and the parser grows from the current lightweight token grammar into a fuller syntax layer closer to upstream `wgsl-parse`. |
| PARITY-002 | `wesl::SourceMapper`, pretty diagnostics in `wesl-rs` | Add span-aware diagnostics instead of message-only errors. | `IN_PROGRESS` | Phase 2. The 2026-04-30 diagnostic slice introduced a public `Diagnostic` / `DiagnosticDetail` / `SourceSpan` / `SourcePosition` layer mirroring the upstream error wrapper shape. Diagnostics now carry module path, resolver display name, declaration, optional source/output, optional line/column span, message override, origin display helpers, source-snippet rendering, and line-span inference from existing compiler context strings. New `compile_diagnostic`, `compile_sourcemap_diagnostic`, and `Wesl::compile_diagnostic` entrypoints preserve the existing `WeslCompileError` API while offering an upstream-style diagnostic path. Remaining gap: parser and validator internals still need to raise structured spans directly instead of relying on item-level spans or message-context inference. Depends on `PARITY-001`. |
| PARITY-003 | `wesl::SourceMapper`, `Wesl::use_sourcemap` | Add source map generation for emitted WGSL. | `IN_PROGRESS` | Phase 2. `CompileResult` already returns an optional `BasicSourceMap`, and the 2026-04-30 slices added the missing reverse lookup surface needed for upstream-style diagnostics: `ManglerKind::unmangle` now supports reversible escape/unicode schemes, `NoSourceMap` mirrors the upstream no-op sourcemap, and `BasicSourceMap` / `CompileResult` can unmangle diagnostic text from generated WGSL identifiers back to `module::decl` names. The newest source-map slice adds a public `SourceMapper[R]` proxy with `Resolver` forwarding, source recording, declaration mangle recording, `finish`, and fallback unmangle behavior. The compiler no longer rebuilds sourcemaps after successful compilation; it records sources and declarations while modules are loaded, which means diagnostic compilation can retain partial source/declaration maps even when import/validation fails mid-pipeline. This is tested through an actual imported declaration compiled with `compile_sourcemap`, a validation failure compiled through `compile_sourcemap_diagnostic`, direct `SourceMapper` black-box behavior, and a white-box missing-import failure that verifies imported source/declaration mappings survive the failure. Remaining gap: route all parser/validator/lower errors through first-class span-bearing diagnostics rather than the current mix of structured maps and legacy `WeslCompileError` variants. |
| PARITY-004 | `StandardResolver`, `FileResolver`, resolver routing in `wesl-rs` | Expand resolver support beyond `VirtualResolver`. | `IN_PROGRESS` | Phase 3. The project now has real `NoResolver`, `VirtualResolver`, `Preprocessor`, `PkgResolver`, `StandardResolver`, and `Router` coverage, plus a filesystem-backed `FileResolver` that reads physical files and follows the upstream default `.wesl` then `.wgsl` fallback. On 2026-04-30 this was verified against vendored Bevy fixtures by compiling `bevy/custom_material.wesl` through `FileResolver`, including its local `super::util` import. The package artifact slice also lets `StandardResolver` consume scanned `Pkg` artifacts directly through `add_scanned_package`. Remaining gap: full Bevy package-import coverage still needs upstream-shaped package artifacts for the real Bevy shader package graph rather than small generated fixtures. |
| PARITY-005 | `wesl::Wesl` high-level API | Add a high-level compiler facade on top of the low-level `compile(...)` function. | `IN_PROGRESS` | Phase 2. The project now exposes an upstream-shaped `Wesl[R]` facade with `new`, `new_experimental`, `new_barebones`, resolver injection, standard resolver package/constants helpers, option/mangler setters, sourcemap toggling, imports/condcomp/generics/strip/lower switches, feature configuration, keep declarations, `compile`, `compile_diagnostic`, and `build_artifact`. Remaining gap: Rust-style custom mangler trait injection is not represented yet because the MoonBit side currently models mangling as `ManglerKind` rather than an object-safe custom strategy. |
| PARITY-006 | `wesl` validation surface | Add validation stages for WESL and generated WGSL. | `IN_PROGRESS` | WESL-specific validation owns declaration duplicates and module/declaration cycles only. `compile(validate=true)` validates final assembled WGSL through an internal adapter over `Milky2018/wgsl/ir`; the former public WESL-owned `validate_wgsl` API and duplicate builtin/type tables are removed. Matrix scalar legality is a WGSL IR invariant. Remaining gap: Core errors currently map to WESL diagnostics without precise source spans. Depends on `PARITY-001`. |
| PARITY-007 | `wesl` lower/eval features | Replace textual lowering with semantic lowering and const-evaluation. | `IN_PROGRESS` | Phase 4, but a first evaluator slice landed early on 2026-04-15 because the vendored upstream spec-eval corpus was already in place and narrow enough to drive real progress. `moon_wesl` now has a dedicated `const_eval.mbt` module with a tokenized `ArrayView` parser/evaluator for a constrained `@const fn test() -> T { ... }` subset used by the upstream literal/type-inference cases. The current evaluator handles `let`/`var` bindings, `return`, unary negation, `+` / `-` / `/`, nested parentheses, `clamp(...)`, abstract-vs-concrete integer inference across `i32` / `u32` / `f32`, decimal and hexadecimal integer literals, decimal and hexadecimal float literals, range checking, and exact `f32` round-trip validation for cases such as `0x1.0000000001p0`. The 2026-05-01 execution slices add upstream-shaped `Inputs`, `ResourceBinding`, `OverrideValue`, and `ExecResult` surface types plus `CompileResult::exec` for simple entrypoints whose body fits the current const-evaluator. `EvalResult::to_buffer()` now exposes upstream-style little-endian scalar and vector buffers for storable `i32`, `u32`, and `f32` values while returning `None` for non-storable values such as `bool`; `ExecResult::to_buffer()` mirrors that behavior for simple entrypoint return values. Entry point parameters annotated with supported `@builtin(...)` attributes are now bound from `Inputs`, including scalar, vector, subgroup, primitive, and view-index builtins exposed by `wesl 0.3.2`; direct `@location` parameters and struct entrypoint parameters with `@builtin` / `@location` members now bind through `Inputs.user_defined`. Validation recognizes the same builtin identifier surface before execution. Pipeline override declarations now participate in the exec context, and the resource slices load scalar/vector `i32` / `u32` / `f32` uniform/storage buffers into global resource declarations by `@group` / `@binding`. The newest layout slices add unit-aware struct, array, and matrix resource/return-buffer conversion using WGSL member alignment, vec3 alignment, member `@align` / `@size` attributes, array stride padding, runtime array length inference from remaining bytes, and column-major matrix padding. The vector semantic slice applies vector arithmetic, comparisons, `clamp`, `min`, `max`, `abs`, `select`, `all`, and `any` component-wise, including scalar broadcast into vector operations. The latest evaluator parser slices add real postfix expression nodes for member access and indexing, so entrypoint structs can be read as `input.field`, vector values support WGSL `xyzw` / `rgba` swizzles, and vector/array/matrix values can be read with dynamic `value[index]` expressions. Statement execution now also has AST-backed `if` / `else`, `while`, `for`, `loop`, and `switch` nodes with local assignment, compound assignment, scoped `break`, `continue`, `continuing`, and `break if` control propagation. The expression parser now distinguishes vector constructor templates from relational `<`, so loop conditions such as `i < limit` no longer get misclassified as generic calls. The 2026-05-05 slices add structured lvalue assignment for root variables, fields, indexes, and vector components; execute void entrypoints and `return;`; share one parser path for `=`, compound assignment, `++`, and `--`; write scalar, field, component, and indexed storage updates back into `ExecResult.resources`; and evaluate `arrayLength(&runtime_array)` against runtime-sized storage arrays inferred from input buffer lengths. Remaining gap: this is not yet the full `wgsl-types` instance/resource execution engine for pointer-style writes, atomics, textures, or arbitrary shader control flow. |
| PARITY-008 | `PkgBuilder`, `build_artifact`, `include_wesl!` workflow | Add a package/build integration story for reusable shader libraries. | `IN_PROGRESS` | Phase 3. The 2026-04-30 package artifact slice introduced MoonBit-native `PkgBuilder`, `Pkg`, and `ScanDirectoryError`. `PkgBuilder::scan_root` now scans `.wesl` / `.wgsl` files plus same-stem submodule directories, normalizes dash-containing package names to underscores, rejects reserved module names, skips invalid identifier filenames, validates scanned module syntax, converts to `CodegenPkg`, and can write a deterministic textual artifact through `Pkg::build_artifact`. `CompileResult::write_to_file`, `CompileResult::write_artifact`, `Wesl::build_artifact`, and `emit_rerun_if_changed` now mirror the upstream build.rs artifact workflow for compiled WGSL outputs, using `OUT_DIR` by default and accepting an explicit MoonBit `out_dir` for tests/build tools. `StandardResolver` and `Wesl` can consume scanned packages via `add_scanned_package`. Tests cover real filesystem scanning, artifact writing, package resolver lookup, compiling a root shader that imports a scanned package through the standard resolver, and facade-driven WGSL artifact generation. Remaining gap: MoonBit-native include/build integration is still textual rather than a first-class build-system artifact macro equivalent, and dependency artifact loading still needs a stronger story. Depends on `PARITY-004` and `PARITY-005`. |

## Current Work Queue

- Phase 1: compiler correctness foundation
  - `PARITY-001`: AST-backed compiler core
  - `PARITY-006`: validation stages
  - `PARITY-012`: parity and regression test suite
- Phase 2: diagnostics and library ergonomics
  - `PARITY-002`: span-aware diagnostics
  - `PARITY-003`: sourcemap support
  - `PARITY-005`: high-level compiler facade
- Phase 3: resolver and ecosystem integration
  - `PARITY-004`: standard/file/router resolvers
  - `PARITY-008`: package/build artifact workflow
- Phase 4: advanced language and execution features
  - `PARITY-007`: semantic lowering and const-eval
  - `PARITY-011`: experimental generics

## Planning Notes

- Recommended strategy: close Phase 1 completely before starting Phase 3 or 4.
- `PARITY-001` is the keystone item. Without it, the project will keep paying
  complexity tax in every other parity area.
- `PARITY-012` should advance in lockstep with every completed issue rather than
  being deferred to the end.
- `PARITY-011` is intentionally late because it adds surface area but does not
  improve the reliability of the current core compiler.
- Vendored Bevy/WGPU corpora are for compiler conformance only. They must not
  drive the core package toward platform/runtime dependencies.
